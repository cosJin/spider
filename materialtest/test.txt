    <h3>Energy Corrections</h3>
  </div>
  <div class='modal-body'>
    <p>
      For more detail, pelase see
      the <a href="/docs/calculations#Total_Energy_Adjustments"> Total
      Energy Adjustments
      </a> section of our Calculations Guide.
    </p>
    
      <h4>MPRelaxSet Potcar Correction</h4>
      <div>Checks that POTCARs are valid within a pre-defined input set. This
    ensures that calculations performed using different InputSets are not
    compared against each other.

    Entry.parameters must contain a &quot;potcar_symbols&quot; key that is a list of
    all POTCARs used in the run. Again, using the example of an Fe2O3 run
    using Materials Project parameters, this would look like
    entry.parameters[&quot;potcar_symbols&quot;] = [&#39;PAW_PBE Fe_pv 06Sep2000&#39;,
    &#39;PAW_PBE O 08Apr2002&#39;].</div>
    
      <h4>MP Gas Correction</h4>
      <div>Correct anion energies to obtain the right formation energies. Note that
    this depends on calculations being run within the same input set.</div>
    
      <h4>MP Anion Correction</h4>
      <div>Correct gas energies to obtain the right formation energies. Note that
    this depends on calculations being run within the same input set.</div>
    
      <h4>MP Advanced Correction</h4>
      <div>This class implements the GGA/GGA+U mixing scheme, which allows mixing of
    entries. Entry.parameters must contain a &quot;hubbards&quot; key which is a dict
    of all non-zero Hubbard U values used in the calculation. For example,
    if you ran a Fe2O3 calculation with Materials Project parameters,
    this would look like entry.parameters[&quot;hubbards&quot;] = {&quot;Fe&quot;: 5.3}
    If the &quot;hubbards&quot; key is missing, a GGA run is assumed.

    It should be noted that ComputedEntries assimilated using the
    pymatgen.apps.borg package and obtained via the MaterialsProject REST
    interface using the pymatgen.matproj.rest package will automatically have
    these fields populated.</div>
